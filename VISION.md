# PIECE — 目指すべき姿と開発サイクル

## Mission

> コードベースに関するあらゆる質問に、嘘なく、根拠付きで、自動で答える。
> 使うほど賢くなり、答えられない質問がなくなっていく。

PIECEはコードを書き換えるツールではない。
**調査・蓄積・検索・回答の精度を極限まで高める、知識のスペシャリスト。**

---

## 理想の定義（品質基準）

### 1. 回答精度

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 全回答にコード根拠あり。ファクトチェック通過率100%。矛盾ゼロ | 未達 |
| A | 90%以上の回答が検証済み。未検証部分は明示される | 概ね到達 |
| B | 回答はできるが、根拠が曖昧な場合がある | - |
| C | 回答に誤りが含まれうる | - |

**目標: 常にS維持。Aに落ちたらギャップ分析して修復。**

### 2. 知識カバレッジ

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 対象コードベースの全関数・全フロー・全設定を知識DBが網羅 | 未達 |
| A | 主要フロー・主要モジュールは網羅。エッジケースに穴がある | 部分的 |
| B | 分析済みだが、知識の深さにムラがある | - |
| C | 分析が浅く、表面的な回答しかできない | - |

**目標: Sに到達し維持。MECE Matrixで網羅性を定量測定。**

### 3. 検索精度

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 質問に対し、最も関連性の高い知識が確実にtop-3に入る。ノイズゼロ | 未達 |
| A | top-5にほぼ正解あり。たまに無関係ノードが混入 | 到達(小規模テストMRR=1.0) |
| B | 関連知識は見つかるが、ランキングが不安定 | - |
| C | 検索漏れが頻繁に発生 | - |

**目標: S維持。6戦略+適応的重み+MMR+LSH-ANNで実現。**

### 4. 自己成長

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 質問・調査・フィードバック全てが知識を強化。概念メッシュが自律成長。矛盾を自動検出・解決 | 未達 |
| A | 質問→知識保存→ヘッブ学習は動作。矛盾検出は手動 | 概ね到達 |
| B | 知識は保存されるが、成長が遅い | - |
| C | 使っても賢くならない | - |

**目標: S。調査ループ・フィードバック・概念自動成長の3軸で実現。**

### 5. ファクトチェック

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 全主張がコード行番号付きで検証済み。プログラム的検証+AI検証の二重チェック。偽陽性ゼロ | 未達 |
| A | AI検証は動作。行番号検証もあるが、一部すり抜けあり | 概ね到達 |
| B | ファクトチェックはあるが信頼性にムラ | - |
| C | ファクトチェックが形骸化 | - |

**目標: S。検証の厳密さと根拠の追跡可能性を極限まで。**

### 6. 回答速度

| レベル | 基準 | 現状 |
|--------|------|------|
| S | キャッシュヒット時は即答(<1秒)。新規質問でも10秒以内 | 未達 |
| A | キャッシュヒットは速い。新規は30秒〜1分 | 概ね到達 |
| B | 毎回AIを呼ぶため遅い | - |
| C | タイムアウトが頻発 | - |

**目標: S。知識の蓄積で「AI呼び出し不要」な回答を増やし続ける。**

### 7. 誠実性

| レベル | 基準 | 現状 |
|--------|------|------|
| S | わからないことは「わからない」と言う。矛盾ゼロ。知ったかぶりゼロ | 未測定 |
| A | 大部分で正直。稀に不確実な情報を断定的に述べる | - |
| B | 不確実性を認めない傾向。矛盾が散見 | - |
| C | 嘘が混入。存在しないファイル・関数を参照 | - |

**目標: S。嘘ペナルティ（存在しないファイル参照=-3点/件）で厳格に監視。**

### 8. フロー追跡精度

| レベル | 基準 | 現状 |
|--------|------|------|
| S | E2Eフローを聞いたら、正確な処理順序を3ステップ以上で記述 | 未測定 |
| A | 主要ステップは記述されるが、順序や詳細に抜けがある | - |
| B | フローの概要は述べるが、具体的なステップ記述が不足 | - |
| C | フローを追跡できない | - |

**目標: S。フロー質問では必ず処理ステップを順序付きで記述。**

### 9. 提案力

| レベル | 基準 | 現状 |
|--------|------|------|
| S | 質問への直接回答+関連情報+注意点+改善提案を含む | 未測定 |
| A | 回答+一部の関連情報 | - |
| B | 質問にしか答えない。付加価値なし | - |
| C | 質問にすら十分に答えられない | - |

**目標: S。顧客の悩みを解決する提案まで踏み込む。**

## ベンチマーク（9軸厳格採点、100点満点）

### 配点（凍結: この配点と質問セットは変更禁止）

```
  1. ファクトチェック通過率   15点 — 検証0件=0点
  2. ファイル正確性           15点 — 嘘1件=-3点
  3. 用語網羅性               10点
  4. 根拠の具体性             15点 — 実在ファイル+実在行番号
  5. 回答の実質性             10点 — 必須ファクト言及率
  6. 誠実性                   10点 — 矛盾=-2点、知ったかぶり減点
  7. フロー追跡精度           10点 — 処理ステップ3以上
  8. 提案力                    5点 — 関連情報・注意点の付加
  9. 応答速度                 10点 — キャッシュ<1秒、初回<20秒
```

### 基準凍結ルール

```
1. 配点の変更は禁止。スコアの経時比較を可能にするため
2. 質問セット(t1-t15)の変更は禁止。同じ問題で測り続ける
3. 採点ロジック(e2e-benchmark.ts)の変更は原則禁止
   - 変更が許されるのはバグ修正のみ
   - 「スコアが上がるような採点変更」は絶対禁止
4. 質問の追加は許可（t16以降）。ただし既存t1-t15は不変
5. 新しいテストターゲット(別のOSS)の追加は許可。別スコアとして記録
```

### 実体改善の原則

```
スコアを上げるためにやっていいこと:
  ✅ スペシャリストのプロンプトを改善して、回答品質を上げる
  ✅ ファクトチェッカーがソースを読み込めるようにパイプラインを修正する
  ✅ 知識DBの検索精度を上げて、より正確な知識を回答に反映させる
  ✅ エージェントの協調を改善して、回答の網羅性を上げる
  ✅ 分析フェーズを改善して、スペシャリストの知識品質を上げる

スコアを上げるためにやってはいけないこと:
  ❌ 採点ロジックを変えてスコアが上がるようにする
  ❌ 質問セットを変えて簡単な質問にすり替える
  ❌ 回答のフォーマットだけ変えて採点パターンに合わせる
     （例: ファイルパスを回答末尾に機械的に追記する）
  ❌ ベンチマーク質問の正解を知識DBにハードコードする

判断基準:
  「もし質問セットを完全に入れ替えても同じスコアが出るか？」
  → Yes ならば実体改善。No ならば水増し。
```

## 開発サイクル（毎セッション実行）

### Phase 0: 情報収集（リサーチ）

ギャップを埋める最良の手段を見つけるため、**実装の前に調べる**。
**毎回異なるテーマを調査すること。前回と同じ検索は禁止。**

```
手順:
  1. Phase 1（現状診断）の結果から、今回最もギャップが大きい領域を特定
  2. その領域に絞って、まだ調べていない切り口で検索する
  3. WebSearch で最新の研究・技術情報を検索
  4. WebFetch で有望な記事・論文の詳細を取得
  5. 得られた知見をPIECEにどう応用できるか評価
  6. 採用する技術・手法を決定し、Phase 2の目標設定に反映

対象領域（毎回ギャップに応じて選択）:
  - 学術論文・研究: 検索、知識表現、NLP、情報検索
  - AI技術: RAG、エージェント設計、ファクト検証、知識グラフ
  - IT技術: SQLite拡張、ベクトル検索、インデックス設計
  - 応用事例: 類似システム（Sourcegraph, Greptile, Bloop等）
  - OSS: 利用可能なライブラリ、アルゴリズム実装

リサーチ重複防止:
  - 調査済みテーマは CHANGELOG.md のセッション記録で確認
  - 前回と同じキーワードで検索しない。別の角度・別の深さで掘る
  - 例: 前回「hybrid search」を調べた → 今回は「query understanding」や
        「relevance feedback」など、検索パイプラインの別の段階を調べる
```

**原則: 闇雲に実装しない。世界中の知見を活用して最善手を打つ。毎回新しい知見を仕入れる。**

### Phase 1: 現状診断

毎セッションの冒頭で以下を実行する。

```
1. git log --oneline -20 で最近の変更を確認
2. 上記6領域の品質基準に対して現状レベルを評価
3. 前回セッションからの進捗を確認
4. 最も大きなギャップ（理想と現状の差）を特定
5. Phase 0の知見を加味し、ギャップの埋め方を再評価
```

**判断基準: どの領域がSから最も遠いか？ どの改善が最もインパクトが大きいか？ 最新技術で解決できるか？**

### Phase 2: 目標設定（自律判断）

**自分で考えて、自分で決める。** 指示待ちではなく、最も効果的な改善を自律的に選択する。

```
選択肢（制限なし。最もインパクトの大きいものを自分で判断）:
  - 新機能・アルゴリズム改善
  - 大規模リファクタ・アーキテクチャ変更
  - バグ修正
  - 設計の根本的な見直し（「このアプローチは間違ってた」も許可）
  - 技術導入（リサーチで見つけた新手法）
  - 統合テスト
  - 上記の組み合わせ

判断基準:
  - 「スコアが最も上がるのはどれか」を自分で考える
  - 3回同じアプローチで改善しなかったら方針転換する（CHANGELOG参照）
  - 小さな修正で済むなら小さく。大規模変更が必要なら恐れずやる
  - ただし安全装置を必ず通す（ビルド・テスト・ベンチマーク）
```

**原則:**
- 何をやるかは自分で決める。ただしスコアの水増しは絶対禁止
- 壊れたらpushしない（ビルド失敗 or テスト失敗 → commit禁止）
- 大きな設計変更をしたらISSUES.mdに「なぜこの判断をしたか」を記録
- 3回連続でスコアが改善しなかったら、根本的にアプローチを変える

### Phase 3: 実装（制限なし）

```
自分で判断して実装する。以下はガイドラインであってルールではない。

- 変更対象のコードを読み、現状を正確に理解する
- 必要なら大規模リファクタも行う
- 必要なら新しいファイルを作る
- 必要なら既存のアーキテクチャを変える
- 必要なら分析パイプライン自体を再設計する

安全装置（これだけは必ず通す）:
  1. npm run build が通ること
  2. npx vitest run tests/search.test.ts が通ること
  3. ベンチマークスコアが前回以下にならないこと（下がったらpushしない）
```

### Phase 4: 振り返り（CHANGELOG.md に追記）

**毎セッション終了時に CHANGELOG.md へセッション記録を追記する。**
これが次回セッションの出発点になる。同じ失敗を繰り返さないための記憶装置。

```
記録フォーマット（CHANGELOG.md に追記）:

## Session YYYY-MM-DD-N（N=その日のセッション番号）

### 診断結果
- 今回のギャップ: （どの領域がSから遠かったか）
- 前回からの変化: （前回の改善は効いたか？ 新たな問題は出たか？）

### リサーチ
- 調査テーマ: （何を調べたか — 次回の重複防止に使う）
- 発見した知見: （使えそうな技術・手法）
- 採用判断: （何を採用し、何を見送ったか。理由も書く）

### 実施内容
- 変更ファイル:
- 何をしたか:
- なぜそうしたか:

### 結果
- ビルド: OK / NG
- 品質基準の変化: （例: 検索精度 B→A）
- うまくいったこと: （次回も続けること）
- うまくいかなかったこと: （次回は避けること、または改善すること）
- 残課題: （時間切れ・スコープ外で持ち越した項目）

### 次回への申し送り
- 最優先ギャップ:
- リサーチ候補: （Phase 0で調べるべきテーマ）
- 注意事項: （踏んだ地雷、避けるべきパターン）
```

**振り返りの原則:**
- 「うまくいった」と「うまくいかなかった」は必ず両方書く
- 「うまくいかなかった」には原因分析を添える（なぜ？ → 次回どうする？）
- 前回の「うまくいかなかった」が今回改善されたか必ず確認する
- 品質基準の「現状」列をVISION.mdで更新する

## 自律検証サイクル（End-to-End品質保証）

開発サイクルの中で、実際のOSSコードベースに対してPIECEを動かし、品質を実測する。

### 仕組み

```
1. テスト対象OSSの選定
   - 中〜大規模のTypeScript/JavaScript OSSを使う
   - 例: Express, Fastify, Next.js, Prisma, tRPC 等
   - 十分な複雑さがあり、フロー追跡・仕様質問のテストに適すること

2. 模擬問い合わせの生成
   - OSSのREADME/docs/コードから、現実的な質問を20〜50個生成
   - カテゴリ: 仕様質問、フロー追跡、設定確認、エラー原因、依存関係
   - 正解（期待される回答の根拠ファイル・行番号）も同時に定義

3. 自動実行
   - npm run dev -- analyze <oss-path>
   - 各模擬質問を npm run dev -- ask <oss-path> "question" で実行
   - 回答を収集

4. 品質評価
   - 回答に含まれるファイルパス・行番号が正解と一致するか
   - ファクトチェック通過率を集計
   - MRR / Recall@K を知識検索部分で測定
   - 「嘘」（存在しないファイル参照、間違った関数名）を検出

5. ギャップ分析→自動改善
   - 品質が基準以下の領域を特定
   - 原因を分析（検索漏れ？ファクトチェック甘い？知識不足？）
   - コードを修正して再テスト

6. 結果をCHANGELOG.mdに記録
```

### 模擬質問テンプレート

```
仕様系:   「<モジュール>の<機能>はどう動く？」
フロー系: 「<操作>を実行したとき、コードはどの順で処理される？」
設定系:   「<設定項目>のデフォルト値は？どこで変更できる？」
エラー系: 「<エラーメッセージ>が出たとき、原因は何が考えられる？」
依存系:   「<ファイル>を変更したら、何に影響がある？」
```

## 技術的工夫の記録（なぜそうしたか）

ここに各改善の工夫と判断理由を蓄積する。後で振り返ったとき「なぜこう設計したか」がわかる宝物。

### 検索システム

| 工夫 | なぜそうしたか | 効果 |
|------|-------------|------|
| LSH-ANN (8平面×3セット) | ベクトル検索がO(n)全走査で50Kノード限界だった。HNSWは実装が複雑すぎる。LSHなら50行で書け、SQLiteに保存でき、95%+リコールが出る。決定論的シードで再現性も確保 | O(n)→O(n/80)、外部依存なし |
| Tag Cluster転置インデックス | 全ノードのタグをGROUP_CONCATで毎回ロードしていた。既存のidx_node_tags_tagインデックスがあるのにLIKEで使えてなかった | O(n×m)→O(candidates) |
| 適応的戦略重み | strategy_performanceテーブルは存在したが誰も読んでなかった。フィードバックデータが無駄になっていた | フィードバックループが閉じた |
| MMR多様化 | 人気ノードが常にtop-kを独占する問題。同じ戦略組み合わせのノードばかり返る | カバレッジ向上、知識空間の探索 |
| Query前処理(NFKC+ファジー) | 全角半角混在で検索漏れ。typoで「authentcation」が何もヒットしない | 検索のロバスト性向上 |

### ファクトチェック

| 工夫 | なぜそうしたか | 効果 |
|------|-------------|------|
| Citation抽出の多パターン対応 | 回答が `[source:path:Lx]` 形式を使わず平文で `src/path/file.ts` と書く。元のextractAllCitationsは[source:]のみ対応で全件ミス | 回答フォーマットに依存しない抽出 |
| フォールバック: specialist docsからソースパス取得 | 回答にソースパスが0件でもファクトチェック不能にしない。specialist docsには[source:]形式の引用がある | ファクトチェック0件→動作する |
| ソースファイルの200行トランケーション | フォールバックで読み込んだファイルが巨大(数千行)だとトークン爆増で3分/問。200行に制限して速度改善 | 180秒→目標20-30秒 |
| ファイル3件上限 | 5件読むと遅い+トークン予算超過。3件で十分なカバレッジ | 速度とコストの最適化 |
| プログラム的検証優先 | AI fact-checkが180秒/問。ファイル存在+行番号確認はプログラムで1秒。AIは「プログラムで3件未満しか検証できなかった場合のみ」呼ぶ | 180秒→1秒(AIスキップ時) |
| 2段階検証戦略 | プログラム的検証(高速・確実)→AI検証(低速・柔軟)の2段階。プログラムで十分ならAI不要 | 速度と品質の両立 |

### ベンチマーク設計

| 工夫 | なぜそうしたか | 効果 |
|------|-------------|------|
| 9軸厳格採点 | 最初の6軸で78点出たが実態は44点。甘い採点は改善を妨げる | 現実の品質が可視化された |
| 嘘ペナルティ(-3点/件) | 存在しないファイルを参照する「嘘」を放置すると信頼性が腐る | 嘘の抑止 |

## 品質原則（不変）

1. **嘘をつかない** — 根拠のない回答は出さない。わからないことは「わからない」と言う
2. **根拠を示す** — 全ての主張にファイルパス・行番号・コードスニペットを付ける
3. **自分を疑う** — ファクトチェッカーは回答者とは独立。自分の回答を自分で検証する
4. **成長する** — 全ての質問・調査・フィードバックが知識を強化する
5. **網羅する** — MECE（漏れなくダブりなく）を追求。知識の穴を自動検出する
6. **速く答える** — 知識があれば即答。AIを呼ばずに済むなら呼ばない

## 現在のアーキテクチャ概要

```
質問 → [Query Cache] → キャッシュヒット → 即答
              ↓ ミス
        [Knowledge DB: 6戦略検索] → 十分な知識あり → DB即答
              ↓ 不足
        [Manager] → Specialist(1-3名) → 各回答
              ↓
        [Fact Checker] → コード突合検証（AI + プログラム的）
              ↓
        [Knowledge Extractor] → 新知識をDB保存
              ↓
        [Mystery Detector] → 未検証→謎として記録
              ↓
        [Investigator] → 謎を自律調査 → 知識追加 → ヘッブ学習
```

**エージェント:** Orchestrator, Specialist, Fact Checker, Investigator, Flow Tracer
**知識基盤:** SQLite + Synapse Engine + Multi-Strategy Search + Atomic Knowledge + MECE Matrix + Concept Mesh
**成長機構:** ヘッブ学習 + 概念自動成長 + フィードバック進化 + 適応的戦略重み
