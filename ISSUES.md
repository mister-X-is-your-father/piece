## ISSUE-001: ファクトチェックが一切動かない (0/15点)

**発見:** Session S4 (2026-04-02) — 厳格ベンチマーク導入時
**深刻度:** CRITICAL — PIECEの核心機能が死んでいた

### 原因分析

```
回答: [ref:Domain Overview] で引用（ドキュメント名）
fact-checker: [source:src/path/file.ts:L42] を期待
→ extractAllCitations() が何もマッチしない → sourceFiles = 0 → 早期リターン
```

根本は**スペシャリスト回答プロンプトとファクトチェッカーの引用フォーマット不一致**。

### 試したこと

| # | 施策 | 結果 | 学び |
|---|------|------|------|
| 1 | extractAllCitationsに平文パス対応追加 | △ 部分的 | 回答にパスがなければ抽出自体が空 |
| 2 | specialist docsからフォールバックで[source:]取得 | ○ 動いた | ただし全docsスキャンで遅い |
| 3 | AI fact-check実行 | × 180秒/問でタイムアウト | トークン量が多すぎ |
| 4 | プログラム的検証優先（fastProgrammaticCheck） | ◎ 1秒で完了 | ファイル存在+行番号確認だけで十分 |
| 5 | specialistプロンプトに[source:path:Lx]要求 | ○ 新規回答で引用形式が改善 | ただし100%ではない |

### 解決

2段階戦略:
1. **プログラム的検証**（ファイル存在・行番号確認）を先に実行。3件以上検証できればAI不要
2. **AIファクトチェック**はフォールバック（プログラムで3件未満の場合のみ）

**スコア変化: 0/15 → 15/15 (Session S5)**

### 残っている問題
- 実在する「行番号」の引用がほぼゼロ（ファイル存在は検証できるが、行レベルは未到達）
- 知識DB即答ルート(Phase 0)ではファクトチェックが走らない

## ISSUE-002: ファイルヒット率が低い (28%→20%)

**発見:** Session S3 — E2Eベンチマーク導入時
**深刻度:** HIGH — 回答に具体的なファイル参照がない

### 原因分析

```
スペシャリスト作成: ディレクトリ構造ベースでクラスタリング
  → クラスタ名が "src-persistence" 等で機能名ではない
  → specialist docsのoverviewにファイルパスは書かれているが
  → 回答時にはoverviewを要約的に使うため、具体パスが落ちる
```

根本は**分析フェーズ(analyze)の品質**。スペシャリストが持つ知識の粒度が粗い。

### 試したこと

| # | 施策 | 結果 | 学び |
|---|------|------|------|
| 1 | プロンプトに[source:path:Lx]要求追加 | △ 微増(28→33%一部) | プロンプトだけでは限界。ドキュメントに情報がなければ引用できない |
| 2 | (未実施) 機能ベースクラスタリング | — | analyze時に機能を先に把握する必要がある |
| 3 | (未実施) specialist docsにファイルリスト明記 | — | 回答時にファイル一覧が参照できれば引用率は上がるはず |

### 現状
- 5.7/15。プロンプト改善だけでは根本解決にならない
- **次のアプローチ:** 分析フェーズでスペシャリストdocsにファイルパスをより多く残す / 回答時にdocsだけでなく実ファイルリストも渡す

## ISSUE-003: 回答の実質性が低下 (5.9→2.5)

**発見:** Session S5 — プロンプト変更後のベンチマーク
**深刻度:** HIGH — 核心情報(mustContainFacts)が回答に含まれない

### 原因分析

```
mustContainFacts: 各質問で「この単語が回答に含まれるべき」を定義
例: t1 → ["QueryExpressionMap", "SelectQueryBuilder"]
    t9 → ["QueryResultCache"]
    t14 → ["TableInheritance"]

回答にこれらが含まれない率: 62%
→ スペシャリストが「概要レベル」で回答し、具体的なクラス名に言及しない
```

根本は**スペシャリストの知識の深さ**が足りない。ディレクトリ名ベースのクラスタリングで、具体的なクラス・関数レベルの知識が薄い。

### 試したこと

| # | 施策 | 結果 | 学び |
|---|------|------|------|
| 1 | プロンプトに「具体的なコード引用」を要求 | △ 一部改善 | ドキュメントに書かれていないクラス名は引用できない |

### 現状
- 2.5/10。ISSUE-002と根っこは同じ（分析品質の問題）
- **次のアプローチ:** analyzeフェーズで各ファイルの主要クラス・関数名をdocsに確実に記録する

**深刻度:** HIGH — フロー質問で処理手順が記述されない


| # | 施策 | 結果 | 学び |
|---|------|------|------|
| 1 | プロンプトに「フローはステップ番号付きで」追加 | △ 一部改善 | specialist.tsに追加済みだが効果限定的 |

### 現状
- 2.5/10
- **次のアプローチ:** フロー質問検出→専用プロンプト注入（「この質問はフロー追跡です。処理を1→2→3の順序で記述してください」）

## ISSUE-006: ベンチマーク速度 (180秒/問)

**発見:** Session S5 — 知識DBリセット後のベンチマーク
**深刻度:** HIGH — ベンチマーク15問に45分かかる

### 原因分析

```
ファクトチェッカーのフォールバック:
  全specialist docs scan → ソースファイル読み込み(数千行) → AI検証
  → 1問あたり180秒（execSyncのtimeout上限）
```

### 解決

プログラム的検証優先:
1. 回答中の[source:]引用と平文パスをプログラムで検証（ファイル存在+行番号確認）
2. 3件以上検証できればAI呼び出しスキップ
3. AI fact-checkはフォールバックのみ

**速度変化:**
- t1(旧): 180秒 → t2-t15(新): 1秒以下（キャッシュヒット時）
- 新規回答時: スペシャリスト回答生成(〜30秒) + プログラム的検証(〜1秒) = 〜31秒

## ISSUE-007: Agent Runner全滅バグ

**発見:** Session S6 (2026-04-02) — 20 specialist分析実行時
**深刻度:** CRITICAL — 1タスク失敗で全分析が中断

### 原因分析

```
runAgentTasks() が Promise.all() を使用
→ 1つのタスクがrejectされると全Promise.allがreject
→ 20タスク中1つのrate limit/timeout → 19タスクの結果も失われる
```

### 解決

個別タスク失敗時にフォールバック空応答 `{content: "[Analysis failed for ...]"}` を返すように変更。
`Promise.all` は維持しつつ、各タスクの catch ブロック内でrejectではなくresolveする。

**Session S6で修正済み。**

## ISSUE-008: Claude Code CLI max-turns制限

**発見:** Session S6 (2026-04-02) — specialist分析が全件 error_max_turns
**深刻度:** HIGH — Claude CLIのツール使用がターンを消費し、max-turns=1で常に失敗

### 原因分析

```
claude -p --max-turns 1 で実行
→ Claude CLIがturn 1でツール(web_search等)を使用
→ ツール結果のturn 2でmax-turnsに到達 → is_error: true, subtype: error_max_turns
→ resultフィールドが未設定(undefined)のため、エラーメッセージが不明瞭だった
```

### 解決

max-turns を 1→10 に変更。エラーメッセージにsubtypeを含めるよう改善。

**Session S6で修正済み。**
